# LLM-TTRA：数据集与铁路环境建模

**文档版本**：2.0  
**日期**：2026-04-27  
**用途**：真实数据集、网络拓扑、列车特性及约束标定的详细说明。对期刊投稿中的可复现性与数据集贡献声明至关重要。

**与架构变更的关联**：本数据集为 UAО-RD 统一Agent编排器、L1-L4工作流及分层求解器提供底层数据支撑。SnapshotBuilder基于此数据集执行时空裁剪，将完整问题（147列×13站）缩减为可计算子问题。

---

## 1. 数据来源与范围

### 1.1 来源

全部时刻表数据来源于**京广高速铁路（京广高铁）实际运营时刻表**。系统当前建模北段：

- **起点**：北京西
- **终点**：安阳东
- **里程**：约500公里
- **走廊**：13个节点、12个区间

### 1.2 规模

| 指标 | 数值 |
|------|------|
| 列车总数 | 147（均为G字头高速动车组） |
| 车站总数 | 13（12座车站 + 1个线路所） |
| 区间总数 | 12 |
| 时间覆盖 | 全日运营时段 |
| 数据文件 | `plan_timetable.csv`、`station_alias.json`、`train_id_mapping.csv`、`min_running_time_matrix.csv` |

### 1.3 数据真实性声明

本数据**非合成数据**。所有到发时间、车次号、车站序列及区间运行时间均来自已发布的运营时刻表。这使LLM-TTRA区别于先前工作，后者通常使用：
- 随机生成的实例
- 简化的小型网络（5-10站）
- 人工构造的延误场景

## 3. 列车数据

### 3.1 列车特性

- **类型**：全部为G字头（高速动车组，CRH/CRH380/CR400系列）
- **服务模式**：长途与中途混合服务
- **径路覆盖**：并非所有列车运行完整13站走廊，部分仅服务子区段
- **停站模式**：大站（SJP、BDD）近乎必停；小站存在部分跳停

### 3.2 数据格式

**`plan_timetable.csv`**（主数据源）：
```csv
train_id,station_1_A,station_1_D,station_2_A,station_2_D,...
1,06:00,06:05,06:15,06:17,...
```

**`train_id_mapping.csv`**：
```csv
numeric_id,train_number
1,G1215
2,G1563
...
```

**`trains.json`**（预处理后的回退）：
```json
{
  "train_id": "G1215",
  "train_type": "G",
  "schedule": {
    "stops": [
      {"station_code": "BJX", "arrival_time": "06:00", "departure_time": "06:05", "stop_duration": 300},
      ...
    ]
  }
}
```

### 3.3 停站模式分析

| 模式 | 占比 | 示例 |
|------|------|------|
| 全程（BJX-AYD） | ~30% | 长途服务 |
| 部分（BJX-SJP 或 BDD-AYD） | ~50% | 中途服务 |
| 短程 | ~20% | 区域接驳 |

### 3.4 高峰时段密度

走廊呈现明显的高峰模式：
- **早高峰（07:00-09:00）**：BJX约15列/小时
- **晚高峰（17:00-19:00）**：约14列/小时
- **平峰（午间）**：约8-10列/小时
- **高峰最小追踪间隔达到**：5-6分钟

## 4. 约束标定

### 4.1 追踪间隔与安全约束

所有约束均依据**CTCS-3（中国列车运行控制系统三级）**标准及运营规章标定。

| 参数 | 数值 | 来源 |
|------|------|------|
| 最小追踪间隔 | 180秒（3分钟） | CTCS-3 350km/h线路设计标准 |
| 最小停站时间 | 120秒（2分钟） | 一般车站下限 |
| 最小停站时间（大站） | 180秒（3分钟） | SJP、BJX运营要求 |
| 多股道发车间隔 | 120秒 | 咽喉区能力约束 |
| 区间压缩上限 | 最小时间的80% | 制动曲线安全裕量 |
| 停站压缩上限 | 原始的50% | 运营可行性 |

### 4.2 冗余参数

来自`config/dispatch_env.yaml`：

```yaml
stop_time_redundancy_ratio: 0.8    # FCFS可用冗余停站时间的80%
running_time_redundancy_ratio: 0.85 # FCFS可用冗余区间运行时间的85%
```

**计算示例**：
- G1215在SJP的原始停站：180秒
- 最小停站：max(120秒, 180秒 × 0.5) = 120秒
- 冗余：180秒 - 120秒 = 60秒
- 可用冗余：60秒 × 0.8 = 48秒

### 4.3 延误分级

| 等级 | 范围 | 响应特征 |
|------|------|---------|
| 微延误 | [0, 5) 分钟 | 常被时刻表缓冲吸收，可能无需主动调整 |
| 小延误 | [5, 30) 分钟 | 需FCFS或快速启发式；受影响列车少时可用MIP |
| 中等延误 | [30, 100) 分钟 | 传播显著；建议使用分层求解器 |
| 大延误 | [100, +∞) 分钟 | 重大扰动；可能需要取消服务或绕行 |

## 5. 延误注入模型

### 5.1 形式化定义

`DelayInjection`由一个或多个`InjectedDelay`记录组成：

```python
class InjectedDelay(BaseModel):
    train_id: str                    # 如 "G1563"
    location: DelayLocation          # 车站或区间
    initial_delay_seconds: int       # 注入点的初始延误
    delay_type: str                  # "primary" 或 "propagated"

class DelayInjection(BaseModel):
    scenario_type: ScenarioType      # 临时限速 / 突发故障 / 区间封锁
    injected_delays: List[InjectedDelay]
    description: str                 # 原始调度员描述
```

### 5.2 传播机制

延误传播**非预计算**；它自然产生于：
1. **初始延误**：加到受影响列车后续所有车站
2. **追踪间隔强制执行**：当延误列车到达某站时，同股道后续列车可能需要等待
3. **级联效应**：每次传播延误都可能触发下游进一步传播

这种涌现性传播是问题组合复杂性的根源。

### 5.3 场景类型

| 类型 | 典型原因 | 默认求解器 | 特点 |
|------|---------|-----------|------|
| 临时限速 | 天气（雨/风）、轨道维护 | MIP | 非紧急；持续时间可预测；精确优化有价值 |
| 突发故障 | 信号故障、接触网问题、设备 | FCFS | 紧急；持续时间不确定；快速响应关键 |
| 区间封锁 | 异物、施工、事故 | FCFS/NoOp | 安全关键；可能需要停运 |

## 6. 数据加载器架构

### 6.1 统一加载接口

```python
# models/data_loader.py

def load_trains() -> List[Dict]: ...
def load_stations() -> List[Dict]: ...
def get_trains_pydantic() -> List[Train]: ...
def get_stations_pydantic() -> List[Station]: ...
def load_min_running_time() -> pd.DataFrame: ...
```

### 6.2 加载策略

1. **主数据源**：`plan_timetable.csv` + `train_id_mapping.csv`
2. **回退**：`trains.json` / `stations.json`（预处理）
3. **车站别名**：`station_alias.json`将中文名映射为3字母代码

### 6.3 Pydantic校验

所有加载数据通过Pydantic模型校验：
- `Train`：`train_id`、`train_type`、`schedule`（`TrainStop`列表）
- `TrainStop`：`station_code`、`station_name`、`arrival_time`、`departure_time`、`is_stopped`、`stop_duration`
- `Station`：`station_code`、`station_name`、`track_count`、`node_type`

校验规则：
- 所有停站满足 `arrival_time <= departure_time`
- 顺序停站满足 `departure_time_i <= arrival_time_{i+1}`
- `stop_duration >= 0`

## 7. 与公开数据集对比

### 7.1 现有铁路调度数据集

| 数据集 | 规模 | 真实世界 | 备注 |
|--------|------|---------|------|
| **LLM-TTRA（本工作）** | 147列、13站 | 是 | 京广高铁北京-安阳段 |
| Rotterdam（荷兰） | ~50列、10站 | 是 | 用于TR Part B论文 |
| HarRail（哈尔滨） | ~200列、20站 | 是 | 中国铁路，部分OR研究使用 |
| SNCF（法国） | 可变 | 是 | 常为发表而简化 |
| 合成（常见） | 10-50列 | 否 | 随机生成，模式不真实 |

### 7.2 本数据集的独特特性

1. **纯高速**：全部为G字头动车组，运行速度300-350km/h；无货运或普速客车混跑
2. **高峰密度极高**：高峰最小追踪间隔5-6分钟，属世界最密集高铁运营之一
3. **混合停站模式**：真实的部分径路服务（非所有列车停靠所有站）
4. **包含线路所**：DJK线路所精确建模，简化模型常忽略此细节
5. **多股道容量真实**：SJP（11股道）准确反映大枢纽容量

## 8. 数据隐私与伦理

### 8.1 数据性质

数据集由**已发布的运营时刻表**组成，属于公开信息。不包含个人数据、专有信号数据或实时运营数据。

### 8.2 局限性

1. **静态时刻表**：代表计划时刻表，非含历史延误数据的实际兑现运营
2. **单走廊**：仅北京-安阳段；完整京广高铁延伸至广州南（约2300公里）
3. **单日**：代表典型日班计划；未捕捉季节性变化

### 8.3 未来数据扩展

计划增强：
- 多日时刻表变化
- 完整京广走廊（北京西—广州南）
- 历史延误记录集成（如可获得）
- 天气关联数据，用于真实场景生成
