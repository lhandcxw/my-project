### 2.2 关键贡献（明确陈述）

1. **首次真实部署**：基于京广高铁实际时刻表的LLM辅助调度（147列、13站）
2. **分层混合求解器**：新型FCFS+MIP组合，动态问题裁剪，相对纯FCFS改进30-60%，求解时间30-60秒
3. **四层LLM-Agent工作流**：L1提取、基于Function Calling的L2策略规划、L3执行、带反射的L4评估
4. **面向高铁的评估**：超越标准延误指标，引入传播控制、公平性（基尼系数）、能耗效率
5. **安全架构**：多层安全兜底，确保LLM失败不会危及运营安全

## 3. 写作风格指南

### 3.1 IEEE T-ITS风格

- **时态**：既定事实用现在时；本文具体工作用过去时
- **语态**：主动语态优先，正式陈述可用被动语态
- **人称**：第三人称；避免过度使用"我们"（IEEE T-ITS偶尔使用"我们"可接受）
- **数字**：1-9拼写；10+用阿拉伯数字
- **公式**：顺序编号；首次使用时定义所有符号

### 3.2 常见错误避免

1. **过度推销LLM能力**：诚实面对LLM局限（幻觉风险、延迟、API依赖）
2. **低估OR贡献**：MIP和分层求解器是重要技术贡献
3. **模糊断言**："显著更好" → "平均总延误减少34.2%（p < 0.01）"
4. **缺失基线**：始终与强基线对比（FCFS、可行时的纯MIP、NoOp）
5. **忽视安全**：在交通领域，安全讨论为强制要求；需回应LLM可靠性关切

### 3.3 贡献陈述关键用语

- "据我们所知，这是首个……的系统"
- "与现有方法不同，后者要么[局限A]要么[局限B]，我们的方法……"
- "核心洞察在于……"
- "该设计动机来自以下观察……"
- "一个显著特征是……"

## 4. 数学符号规范

### 4.1 推荐符号表（论文中应包含）

| 符号 | 含义 |
|------|------|
| $\mathcal{T}$ | 列车集合，$|\mathcal{T}| = n$ |
| $\mathcal{S}$ | 车站集合，$|\mathcal{S}| = m$ |
| $t_i$ / $s_j$ | 列车 $i$ / 车站 $j$ |
| $a_{ij}, d_{ij}$ | 计划到发时间 |
| $A_{ij}, D_{ij}$ | 实际到发时间（决策变量） |
| $\Delta_{ij}$ | 列车 $i$ 在车站 $j$ 的延误 |
| $h$ | 最小追踪间隔（180秒） |
| $\tau^{\min}_{j,j+1}$ | 最小区间运行时间 |
| $c_j$ | 车站 $j$ 的股道数 |

### 4.2 公式排版

多行约束使用`align`环境。示例：

```latex
\begin{align}
\min \quad & \sum_{i \in \mathcal{T}} \sum_{j \in \mathcal{S}_i} (\Delta^{\text{arr}}_{ij} + \Delta^{\text{dep}}_{ij}) \\
\text{s.t.} \quad & A_{ij} \geq a_{ij} + \delta_{ij}, && \forall (i,j) \in \mathcal{D} \\
& A_{i,j+1} - D_{ij} \geq 0.8 \tau^{\min}_{j,j+1}, && \forall i, j \\
& D_{kj} - D_{ij} \geq h, && \forall \text{同股道相邻 } (i,k) \\
& D_{ij} - A_{ij} \geq \sigma^{\min}_j, && \forall i, j \text{（停车）}
\end{align}
```

## 7. 审稿预判与回应

### 7.1 可能关切与先发制人回应

**Q1："为何用LLM而非基于规则的专家系统？"**
> A：规则系统无法处理自然语言歧义或新场景。LLM可从口语化调度员语言中提取结构化信息（如"大概延误十多分钟"）。但安全关键决策（区间封锁处理）由硬规则兜底保护。

**Q2："若LLM幻觉或选择错误求解器怎么办？"**
> A：三层安全机制：(1) L3覆盖危险选择（区间封锁必须用FCFS），(2) 策略引擎拒绝超限方案，(3) ARDO可带反馈重规划最多3轮。v8.0中`FORCE_LLM_MODE=true`禁用规则回退，以收集纯LLM性能数据。

**Q3："分层求解器是否可证明最优？"**
> A：否，且我们不声称最优。分层求解器是权衡最优性与可扩展性的近似算法。裁剪窗口上的MIP对该窗口最优，FCFS提供全局兜底。我们显式报告最优性间隙。

**Q4："为何不用RL代替LLM进行求解器选择？"**
> A：RL需要大量环境交互与奖励塑造。当前方法利用LLM的预训练推理能力配合少样本提示。我们正收集SFT数据（`data/sft_train.jsonl`）用于未来RL微调，详见未来工作章节。

**Q5："能否推广至其他铁路网络？"**
> A：系统由车站拓扑、股道数、时刻表数据参数化。任何具有兼容数据格式的走廊均可加载。京广高铁数据集验证了全球最密集高铁走廊之一上的可行性。
